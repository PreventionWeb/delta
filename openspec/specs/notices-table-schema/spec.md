## ADDED Requirements

### Requirement: noticesTable declares the correct columns and types

`noticesTable` SHALL be exported from `app/drizzle/schema/noticesTable.ts` as a Drizzle
`pgTable` named `"notices"`. It MUST contain the following columns with the specified
types and constraints:

- `id` — UUID primary key, default `gen_random_uuid()`, NOT NULL.
- `countryAccountsId` — UUID, NOT NULL, foreign key referencing `countryAccountsTable.id`
  with `onDelete: "cascade"`.
- `titleJson` — JSONB, NOT NULL (a notice must always have a title; drafts use `isPublished = false`, not missing fields).
- `bodyJson` — JSONB, nullable (no default).
- `isPublished` — boolean, NOT NULL, default `false`.
- `publishedAt` — TIMESTAMPTZ, nullable (no default).
- `audience` — text enum constrained to `('public', 'private', 'all')`, NOT NULL,
  default `'private'`.
- `createdAt` — TIMESTAMPTZ, NOT NULL, default `CURRENT_TIMESTAMP`.
- `updatedAt` — TIMESTAMPTZ, default `CURRENT_TIMESTAMP`.

The file MUST also export `SelectNotice` (inferred from `noticesTable.$inferSelect`) and
`InsertNotice` (inferred from `noticesTable.$inferInsert`).

#### Scenario: Insert a minimal draft notice (only required fields)

- **GIVEN** a valid `countryAccountsId` exists in the database
- **WHEN** a row is inserted into `notices` with only `countryAccountsId` provided
- **THEN** the inserted row SHALL have `isPublished = false`, `audience = 'private'`,
  `publishedAt = null`, `bodyJson = null`, and non-null values for `id`, `createdAt`,
  and `titleJson` (which must be supplied — it is NOT NULL)

#### Scenario: Insert a fully populated notice

- **GIVEN** a valid `countryAccountsId` exists in the database
- **WHEN** a row is inserted with `titleJson`, `bodyJson`, `isPublished = true`,
  `publishedAt` set to a specific timestamp, and `audience = 'public'`
- **THEN** the retrieved row SHALL reflect all supplied values without modification

#### Scenario: `audience` column declares the correct enum members

- **WHEN** the `noticesTable.audience.enumValues` property is inspected
- **THEN** it SHALL equal `["public", "private", "all"]` — confirming the Drizzle column
  definition carries the enum metadata.
  Note: per design.md Decision 2, the codebase uses `text({ enum: [...] })` rather than
  `pgEnum` due to a known drizzle-kit bug (drizzle-team/drizzle-orm#3485). Drizzle's
  `text({ enum: [...] })` stores the enum values as column metadata and surfaces them as
  a TypeScript union type, but does not generate a SQL CHECK constraint. Enforcement is
  at the Drizzle schema / TypeScript level. The test asserts `enumValues` to confirm the
  enum metadata is wired correctly.

#### Scenario: Deleting the parent countryAccountsId cascades to notices

- **GIVEN** one or more notice rows exist for a given `countryAccountsId`
- **WHEN** the corresponding `country_accounts` row is deleted
- **THEN** all associated notice rows SHALL be deleted automatically (cascade)

### Requirement: noticesTable is exported from the schema barrel

The schema barrel `app/drizzle/schema/index.ts` MUST re-export everything from
`./noticesTable` so that `import { noticesTable } from "~/drizzle/schema"` resolves
without error throughout the application.

#### Scenario: Barrel export resolves the table

- **WHEN** `noticesTable` is imported via `~/drizzle/schema`
- **THEN** the import SHALL resolve to the same `noticesTable` object defined in
  `noticesTable.ts` without a module resolution error

### Requirement: Migration applies cleanly to a fresh database

The SQL migration generated by `yarn dbsync` for this change MUST apply to a clean
PostgreSQL-compatible database (including PGlite) without error.

#### Scenario: Migration creates the notices table

- **WHEN** `yarn dbsync` is run against a database that does not yet contain the
  `notices` table
- **THEN** the migration SHALL execute without error and the `notices` table SHALL
  exist with the columns, types, defaults, and constraints described above

#### Scenario: Applying the migration twice is idempotent via drizzle-kit journal

- **WHEN** `yarn dbsync` is run a second time after the migration has already been applied
- **THEN** drizzle-kit SHALL detect the migration as already applied via the journal and
  SHALL NOT attempt to re-execute the DDL

### Requirement: PGlite test schema mirror registers the notices table

`tests/integration/db/testSchema/noticesTable.ts` MUST export a schema definition
identical to `app/drizzle/schema/noticesTable.ts` so that `pushSchema` in
`tests/integration/db/setup.ts` creates the `notices` table in the in-memory PGlite
instance. The testSchema barrel `tests/integration/db/testSchema/index.ts` MUST also
re-export this file.

#### Scenario: PGlite setup creates the notices table

- **WHEN** the PGlite test suite initialises via `setup.ts` (which calls `pushSchema`)
- **THEN** the `notices` table SHALL exist in the in-memory database and accept inserts
  against all columns declared in `noticesTable`

#### Scenario: Concurrent insert from two callers sees both rows

- **GIVEN** the PGlite `notices` table exists
- **WHEN** two insert operations are issued concurrently (Promise.all) each providing a
  distinct `countryAccountsId` and distinct row data
- **THEN** both rows SHALL be persisted and both results SHALL be returned without
  conflict or data loss — the table MUST NOT serialise concurrent inserts incorrectly
