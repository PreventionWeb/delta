## Why

The Notices Pilot is a synthetic domain introduced to validate the full DELTA Clean
Architecture stack end-to-end. Phase 4a adds the Drizzle schema and database migration
for the `notices` table — the persistence foundation that every subsequent phase (model,
repository, use-case, route) depends on. Without this table, no notice can be stored or
retrieved.

## What Changes

- **New file** `app/drizzle/schema/noticesTable.ts` — Drizzle `pgTable` definition for the
  `notices` table with all columns specified in the intent: `id` UUID PK,
  `countryAccountsId` UUID FK (tenant isolation, cascade delete), `titleJson` JSONB,
  `bodyJson` JSONB, `isPublished` boolean, `publishedAt` TIMESTAMPTZ nullable,
  `audience` text enum (`'public'|'private'|'all'`) default `'private'`, and
  `createdAt`/`updatedAt` declared inline as TIMESTAMPTZ with `{ withTimezone: true }`
  (NOT via `createdUpdatedTimestamps` — that helper produces plain TIMESTAMP and violates
  ADR-002; see design.md Decision 5).
  Exports `SelectNotice` and `InsertNotice` inferred types.

- **Modified file** `app/drizzle/schema/index.ts` — add `export * from "./noticesTable"`
  so the table is available via the schema barrel.

- **Generated file** `app/drizzle/migrations/<timestamp>_add_notices_table.sql` — SQL DDL
  produced by `yarn dbsync`. Not hand-written; generated and committed as an artifact.

- **Generated file** `app/drizzle/migrations/meta/_journal.json` — updated by drizzle-kit
  with the new migration entry.

- **New file** `tests/integration/db/testSchema/noticesTable.ts` — thin re-export from
  the app schema (`export * from "~/drizzle/schema/noticesTable"`). This is the P1-42
  pilot: instead of duplicating the table definition, the testSchema entry re-exports it.
  The test setup calls `pushSchema` on the testSchema index, so this entry is still
  required for the PGlite suite to create the `notices` table.

- **Modified file** `tests/integration/db/testSchema/index.ts` — add
  `export * from "./noticesTable"` so the PGlite test suite sees the table.

No route, model, handler, or fieldsDef pipeline is touched by this change.

## Capabilities

### New Capabilities

- `notices-table-schema`: The `noticesTable` Drizzle schema — that it declares the
  correct columns with correct types, defaults, and constraints; that it is scoped to a
  tenant via `countryAccountsId`; that `audience` is a text enum with exactly three
  values; and that the migration generated by `yarn dbsync` applies cleanly to a fresh
  database.

### Modified Capabilities

(none — no existing spec-level behaviour changes)

## Impact

- **`app/drizzle/schema/noticesTable.ts`** — new file; Drizzle table definition.
- **`app/drizzle/schema/index.ts`** — modified; adds one `export *` line.
- **`app/drizzle/migrations/<timestamp>_add_notices_table.sql`** — generated by
  `yarn dbsync`; must be reviewed and committed alongside the schema file.
- **`app/drizzle/migrations/meta/_journal.json`** — updated by drizzle-kit.
- **`tests/integration/db/testSchema/noticesTable.ts`** — new file; thin re-export (P1-42 pilot).
- **`tests/integration/db/testSchema/index.ts`** — modified; adds one `export *` line.
- **DB migration**: required. `yarn dbsync` generates the migration; never `drizzle-kit push`.
- **Test approach**: PGlite (`yarn test:run2`). The test verifies the migration applies
  cleanly and the table has the correct columns and constraints. No real PostgreSQL instance
  is needed.
- **Security / multi-tenancy**: `countryAccountsId` FK is `notNull().references(...)` with
  `onDelete: "cascade"` — the same pattern used by all other tenant-scoped tables. No new
  auth surface is introduced.
- **fieldsDef / Form-CSV-API pipeline**: not impacted — no model file is created in this
  phase. The notices domain model comes in a later intent.
