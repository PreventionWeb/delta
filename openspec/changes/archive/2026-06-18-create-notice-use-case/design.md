## Context

The Notices domain follows Clean Architecture. The domain layer (`Notice` entity, `INoticeRepository` port) is complete on branch `feature/ca-notice-entity-and-port`. The application layer is the next ring outward: it hosts use-cases that orchestrate domain objects and ports, and DTOs that carry data across the boundary to the presentation layer.

`Notice.create()` requires a fully-populated `NoticeProps` object including generated fields (`id`, `createdAt`, `updatedAt`) and business-defaulted fields (`audience`, `publishedAt`). The use-case is responsible for generating these before calling the factory.

## Goals / Non-Goals

**Goals:**
- Define `CreateNoticeCommand` — the input value object accepted by the use-case.
- Define `NoticeDto` — the serialisable output returned across the application boundary, with all timestamps as ISO strings (TIMESTAMPTZ-safe).
- Define `toNoticeDto(notice: Notice): NoticeDto` — a pure mapper function.
- Define `CreateNoticeUseCase` — a class with constructor-injected `ILogger` and `INoticeRepository` and a single `execute(command): Promise<NoticeDto>` method.
- Provide a complete unit test file covering happy path, `ValidationError` propagation, and repository error propagation.

**Non-Goals:**
- No route or controller wiring — the use-case is not yet connected to any HTTP handler (Phase 4e or later).
- No Drizzle schema changes — the `notices` table already exists.
- No PGlite or real-DB test infrastructure — this tier is pure unit tests.
- No `audience` selection in `CreateNoticeCommand` — the use-case defaults `audience` to `"private"` in Phase 4d; configurable audience is deferred to a later phase.

## Decisions

### Decision 1: `CreateNoticeCommand` does not include `id`, `createdAt`, `updatedAt`, `audience`, or `publishedAt`

**Rationale:** These fields are not provided by callers. `id` is a UUID generated by the use-case (`crypto.randomUUID()`). `createdAt` / `updatedAt` are set to `new Date()` at creation time. `audience` defaults to `"private"` — this matches the DB column default and is the safe choice for new notices; configurable audience is a future concern. `publishedAt` is `new Date()` when `isPublished` is true, `null` when false — this mirrors the invariant enforced by `Notice.create()`.

**Alternative considered:** Accept `id` from the caller (caller-supplied IDs). Rejected because the use-case owns entity construction; callers should not dictate persistence keys.

### Decision 2: `toNoticeDto()` is a standalone exported function, not a method on the entity

**Rationale:** Entities in Clean Architecture must not know about DTO shapes — DTOs are an application-layer concern. A standalone mapper function in `NoticeDto.ts` keeps the domain layer pure and makes the mapper independently testable.

**Alternative considered:** A `.toDto()` method on `Notice`. Rejected — it would violate the domain/application boundary.

### Decision 3: All `Date` fields in `NoticeDto` are `string` (ISO 8601)

**Rationale:** DTOs cross the application-to-presentation boundary. At that boundary, dates must be serialisable to JSON without information loss. ISO 8601 strings are TIMESTAMPTZ-aware (they include timezone offset). Callers that need a `Date` object can parse the string.

**Alternative considered:** `Date` objects in the DTO. Rejected — `Date` objects are not JSON-serialisable, which breaks the contract when the DTO is eventually sent over HTTP or stored in session.

### Decision 4: `ValidationError` from `Notice.create()` propagates unmodified

**Rationale:** The use-case does not catch or wrap `ValidationError`. The presentation layer (route action) is responsible for catching domain errors and mapping them to HTTP responses. Wrapping at the use-case level would obscure the original error code and message.

### Decision 5: Repository errors propagate unmodified

**Rationale:** Same reasoning as Decision 4. The use-case has no recovery strategy for persistence failures; surfacing them lets the presentation layer decide how to respond (retry, 503, etc.).

### Decision 6: `ILogger` injected — not `console`

**Rationale:** ADR-004. Use-cases must not import `console` directly. The `ILogger` interface is the abstraction; `NoOpLogger` is used in tests. The use-case logs an `info` event after a successful save.

## Risks / Trade-offs

- [Risk] `crypto.randomUUID()` is available in Node 14.17+ and modern browsers. DELTA runs Node 18+, so this is safe. If a future test environment lacks it, a polyfill must be added.
  → Mitigation: DELTA's `package.json` `engines` field documents the Node version requirement — no inline comment needed in the source file.

- [Risk] `audience` defaults to `"private"` — if a future command includes an `audience` field, the default must be removed and the spec updated.
  → Mitigation: the default is explicit and documented in code; a future spec delta will cover it.

- [Risk] `createdAt` / `updatedAt` are set to `new Date()` inside the use-case. Tests that need deterministic timestamps must inject a known date by passing it directly — the current design does not support clock injection.
  → Mitigation: unit tests can use `vi.setSystemTime()` from Vitest to freeze time; this is idiomatic and avoids over-engineering a clock dependency.

## Test Infrastructure

Unit tests only. No PGlite, no database. The test file is:

`app/domains/notices/application/use-cases/CreateNotice.test.ts`

Mock strategy:
- `INoticeRepository`: inline `vi.fn()` mock object — stub `save` to resolve with the input entity or reject with an error as needed. Never use `as any`; type the mock as `INoticeRepository` explicitly using a helper object that satisfies the interface.
- `ILogger`: `NoOpLogger` instance imported from `app/shared/logging/NoOpLogger.ts`.
- Time: `vi.useFakeTimers()` / `vi.setSystemTime()` in tests that assert timestamp values.
